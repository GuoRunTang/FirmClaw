## 一、项目概览

### 1.1 FirmClaw 是什么

FirmClaw 是一个从零构建的 AI 智能体框架，核心能力是：**让 LLM 通过调用工具来与操作系统交互，自主完成复杂任务**。

### 1.2 技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| 语言 | TypeScript | 类型安全，生态丰富 |
| LLM | MiniMax M2.7（OpenAI 兼容格式） | 1M 上下文窗口，支持 function calling |
| SDK | openai（官方 OpenAI SDK） | 一套代码兼容所有 OpenAI 格式的提供商 |
| 运行时 | Node.js 18+ / tsx | 直接运行 TypeScript，无需预编译 |
| 环境变量 | dotenv | 管理敏感配置，不进入版本控制 |

## 三、核心模块详解

### 3.1 工具系统 — 智能体的"手和脚"

#### 3.1.1 类型定义 (`src/tools/types.ts`)

定义了所有工具必须遵守的接口契约：

```typescript
// 工具的完整定义
export interface Tool {
  name: string;         // 工具名，LLM 通过这个名调用
  description: string;  // 工具描述，告诉 LLM 这工具能做什么
  parameters: ToolDefinition;  // JSON Schema 格式的参数定义
  execute: ToolExecuteFn;      // 实际执行逻辑
}

// 工具执行结果
export interface ToolResult {
  content: string;   // 输出内容，会反馈给 LLM
  isError?: boolean; // 是否为错误，LLM 看到错误会调整策略
}
```

**设计理念**：任何工具（bash、文件读写、网络请求...）只要实现 `Tool` 接口，就能被 Agent Loop 调用。新增工具零耦合。

#### 3.1.2 Bash 工具 (`src/tools/bash.ts`)

智能体与操作系统交互的唯一桥梁：

```typescript
export const bashTool: Tool = {
  name: 'bash',
  description: 'Execute a bash command in the terminal...',
  parameters: {
    type: 'object',
    properties: { command: { type: 'string', description: '...' } },
    required: ['command'],
  },
  async execute(params) {
    // 1. 参数校验
    // 2. child_process.exec 执行命令
    // 3. 捕获 stdout + stderr
    // 4. 错误不抛异常，返回错误信息给 LLM
  }
}
```

**关键设计**：
- **timeout 30s**：防止命令卡死导致智能体挂起
- **maxBuffer 1MB**：防止命令输出撑爆内存
- **错误不中断**：`try/catch` 捕获所有异常，返回错误文本给 LLM，让它自己调整策略

#### 3.1.3 工具注册中心 (`src/tools/registry.ts`)

管理所有工具的注册和查询：

```typescript
export class ToolRegistry {
  private tools: Map<string, Tool> = new Map();
  
  register(tool: Tool): void { ... }   // 注册工具
  get(name: string): Tool | undefined  // 按名查找
  getAll(): Tool[]                      // 获取全部（给 LLM 看）
  toOpenAITools(): OpenAITool[]         // 转为 API 格式
}
```

**为什么需要 Registry**：Agent Loop 不需要硬编码工具列表。注册新工具只需 `registry.register(newTool)`，零改动 Agent Loop。

### 3.2 LLM Client — 与大模型通信的桥梁 (`src/llm/client.ts`)

#### 核心职责

1. 封装 OpenAI 兼容 API 的调用细节
2. 统一内部 Message 格式（屏蔽不同提供商的差异）
3. 支持流式和非流式两种模式

#### 内部消息格式

```typescript
export interface Message {
  role: 'system' | 'user' | 'assistant' | 'tool';
  content: string;
  tool_call_id?: string;     // tool 消息必须带，关联到对应的工具调用
  tool_calls?: ToolCall[];   // assistant 消息可能带，表示要调哪些工具
}

export interface ToolCall {
  id: string;
  type: 'function';
  function: { name: string; arguments: string };  // arguments 是 JSON 字符串
}
```

四种 role 的含义：

| role | 来源 | 包含内容 |
|------|------|----------|
| `system` | 代码注入 | 系统提示词（定义智能体身份和行为规则） |
| `user` | 用户输入 | 用户的消息 |
| `assistant` | LLM 回复 | 纯文本 或 工具调用请求 |
| `tool` | 工具执行结果 | 命令输出、文件内容等 |

#### 流式处理的关键逻辑

OpenAI 的流式工具调用会分多次发送，需要逐步累积：

```
chunk 1: { tool_calls: [{ index: 0, id: 'call_123', function: { name: 'bash', arguments: '' } }] }
chunk 2: { tool_calls: [{ index: 0, function: { arguments: '{"co' } }] }
chunk 3: { tool_calls: [{ index: 0, function: { arguments: 'mmand":"ls"}' } }] }
```

代码中通过 `toolCalls[idx].function.arguments += ...` 逐步拼接，最终得到完整的 `{"command":"ls"}`。

### 3.3 ★ ReAct 循环 — 系统的心脏 (`src/agent/agent-loop.ts`)

#### 3.3.1 什么是 ReAct

**Re**asoning + **Act**ing = ReAct。核心思想是让 LLM 交替进行"思考"和"行动"，直到完成任务。

类比人类解决问题：
1. 看到问题 → 思考方案（Reasoning）
2. 执行方案（Acting）
3. 看到结果 → 继续思考 → 继续行动
4. 直到问题解决

#### 3.3.2 算法伪代码

```
function run(userMessage):
    messages = [系统提示词, 用户消息]
    turns = 0
    toolCallCount = 0

    while turns < maxTurns:
        turns++

        // === Step 1: 调用 LLM（Reasoning） ===
        response = LLM.chat(messages, tools)

        // 记录 LLM 回复
        messages.push(response)

        // === Step 2: 判断响应类型 ===
        if response 没有 tool_calls:
            // LLM 直接回答了 → 任务完成
            return response.content

        // === Step 3: 执行工具（Acting） ===
        for each toolCall in response.tool_calls:
            // 3a. 解析参数
            args = JSON.parse(toolCall.function.arguments)

            // 3b. 查找工具
            tool = registry.get(toolCall.function.name)
            if tool 不存在:
                messages.push({ role: 'tool', content: '错误：未知工具' })
                continue

            // 3c. 执行工具
            result = tool.execute(args)
            toolCallCount++

            // 3d. 把结果反馈给 LLM（作为 observation）
            messages.push({ role: 'tool', content: result.content, tool_call_id: toolCall.id })

        // 循环回到 Step 1，LLM 看到工具结果后继续推理

    return "超过最大轮次限制"
```

#### 3.3.3 一次完整循环的消息流示例

用户问："当前目录有哪些文件？"

```
messages 数组随循环的变化：

初始状态:
  [0] system:   "你是一个本地 AI 智能体助手..."
  [1] user:     "当前目录有哪些文件？"

═══ 第 1 轮循环 ═══

  LLM 看到：用户想知道目录内容 → 需要用 bash 执行 ls
  LLM 返回:
  [2] assistant: { content: "我来查看当前目录的文件", tool_calls: [{ name: "bash", arguments: '{"command":"ls"}' }] }

  Agent Loop 执行 bash("ls") → 得到结果
  [3] tool: { tool_call_id: "call_xxx", content: "docs/  src/  package.json  tsconfig.json" }

═══ 第 2 轮循环 ═══

  LLM 看到：ls 的输出结果 → 可以回答了
  LLM 返回:
  [4] assistant: { content: "当前目录包含以下文件：docs/、src/、package.json、tsconfig.json" }

  没有 tool_calls → 循环结束 → 返回文本给用户

最终 messages 长度: 5 条（system + user + assistant + tool + assistant）
```

#### 3.3.4 关键设计决策

| 决策 | 理由 |
|------|------|
| **maxTurns 限制** | 防止 LLM 陷入无限循环（如反复执行同一命令） |
| **错误不中断** | 工具失败时把错误文本反馈给 LLM，让它自行调整策略，而不是直接崩溃 |
| **tool_call_id 关联** | LLM 可能一次返回多个 tool_calls，通过 id 把每个结果精确对应回请求 |
| **事件广播** | Agent Loop 不直接 console.log，通过 EventStream 通知外部，展示层可自由选择展示方式 |

### 3.4 事件流 — 智能体的"神经系统" (`src/utils/event-stream.ts`)

基于 Node.js 原生 `EventEmitter` 的发布/订阅系统：

```
事件类型:
  thinking_delta  → LLM 正在生成文本（逐字输出）
  tool_start      → 开始执行工具（显示工具名和参数）
  tool_end        → 工具执行完成（显示结果）
  message_end     → 最终回复完成
  error           → 出错了
```

**为什么不用 console.log**：因为 Agent Loop 可能被 CLI、Web UI、测试框架、甚至另一个 Agent 调用，不同场景需要不同的展示方式。事件模式实现了完全解耦。

### 3.5 CLI 入口 — 用户交互界面 (`src/index.ts`)

启动流程：

```
1. 加载 .env 环境变量
2. 读取 LLM_API_KEY / LLM_BASE_URL / LLM_MODEL
3. 创建 LLMClient 实例
4. 创建 ToolRegistry，注册 bashTool
5. 创建 AgentLoop，配置系统提示词和 maxTurns
6. 订阅事件流（thinking → stdout，tool_start → 打印工具名，...）
7. 启动 readline 命令行循环，等待用户输入
```

每次用户输入后的处理：
```
用户输入 → agent.run(input) → 实时展示事件 → 输出统计（轮次、工具调用次数）→ 等待下次输入
```

## 四、.env 与系统环境变量的关系

`dotenv` 库的加载规则：**`.env` 中的变量不会覆盖已存在的系统环境变量**。

优先级（从高到低）：
```
系统环境变量（Windows 环境变量 / PowerShell $env:） > .env 文件 > 代码中的默认值
```

当前使用的环境变量：
| 变量名 | 用途 | 默认值 |
|--------|------|--------|
| `LLM_API_KEY` | API 密钥 | 无（必须配置） |
| `LLM_BASE_URL` | API 基础 URL | `https://api.minimax.chat/v1` |
| `LLM_MODEL` | 模型名称 | `MiniMax-M2.7` |

OpenClaw 技术原理与开发方法
从 ReAct 架构到实际工程实践——理解 FirmClaw 背后的设计哲学

一、什么是 AI Agent？

传统 LLM（大语言模型）只能"对话"——你问它答，回答完了就结束。而 AI Agent（智能体） 不仅能对话，还能主动行动：读文件、执行命令、搜索代码、修改代码、提交 Git 等等。

核心区别：传统 LLM = "嘴巴"（只能说）；AI Agent = "大脑 + 手脚"（能想、能做）。

打个比方：如果你让 ChatGPT "帮我修一下这个 Bug"，它只能给你一段代码建议，你自己去改。但如果你让 Claude Code 或 FirmClaw 修 Bug，它会：

读取你的代码文件

用 read 工具打开相关源文件

分析问题并制定方案

理解代码逻辑，定位 Bug 位置

直接修改代码

用 edit 工具精确修改代码

运行测试验证

用 bash 工具执行测试命令，确认修复

二、ReAct 架构——Agent 的核心原理

ReAct（Reasoning + Acting）是目前 AI Agent 最主流的架构。OpenClaw、Claude Code、FirmClaw 都采用了这个模式。

┌─────────────────────────────────────────────────────────────┐ │ ReAct 循环 (Agent Loop) │ │ │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ Reason │ ──→ │ Act │ ──→ │ Observe │──┐ │ │ │ (思考) │ │ (行动) │ │ (观察) │ │ │ │ └──────────┘ └──────────┘ └──────────┘ │ │ │ ↑ │ │ │ └──────────────────────────────────────────┘ │ │ 循环直到任务完成 │ └─────────────────────────────────────────────────────────────┘

2.1 Reason（思考）

LLM 接收当前上下文（用户需求 + 历史对话 + 工具返回的结果），思考下一步该做什么。

例如：LLM 看到用户说"帮我列出文件"，它会思考："用户想看目录内容，我需要执行 ls 命令"。

2.2 Act（行动）

LLM 不是直接回复文字，而是发起一个工具调用。比如调用 bash 工具执行 ls 命令。

2.3 Observe（观察）

工具执行完毕后，结果（如文件列表）被反馈给 LLM。LLM 看到结果后进入下一轮思考——要么继续行动，要么给出最终回复。

关键洞察：Agent Loop 本质上就是一个 while 循环，里面不断交替执行"调用 LLM 思考"和"执行工具行动"。整个系统的复杂度来自于这个简单循环上的层层增强。

用 Python 思维理解 Agent Loop

如果你熟悉 Python，可以用伪代码这样理解 Agent Loop 的核心逻辑：

# Agent Loop 的核心就是一个 while 循环 while True: # 1. 调用 LLM 思考 response = llm.chat(messages, tools=["bash", "read", "write", "edit"]) # 2. 如果 LLM 决定调用工具 if response.has_tool_calls: for tool_call in response.tool_calls: # 3. 执行工具 result = execute_tool(tool_call.name, tool_call.args) # 4. 把结果反馈给 LLM（Observation） messages.append(tool_result(result)) continue # 继续循环 else: # 5. LLM 没有要调用的工具了，任务完成 print(response.text) break

（上面的伪代码展示了 Agent Loop 最核心的逻辑，实际的 FirmClaw/TypeScript 实现在结构上与此完全一致）

三、工具系统——Agent 的"手脚"

工具系统决定了 Agent 能做什么。OpenClaw 采用"极简主义"——只用 4 个核心工具就覆盖了绝大多数编程场景：

工具功能Python 类比关键设计
bash执行终端命令subprocess.run()流式输出、超时控制、工作目录
read_file读取文件open(f).read()分段读取、二进制检测、带行号
write_file创建/覆写文件open(f,'w').write()自动创建父目录
edit_file精确编辑str.replace()唯一性校验（old_str 只能出现一次）

设计哲学："少量工具 + LLM 强大的理解力 > 大量工具"。这 4 个工具的组合足以让 Agent 完成几乎所有编程任务。

四、会话管理——记忆的骨架

Agent 需要记住之前的对话。OpenClaw 使用 JSONL 格式存储对话历史。

JSONL 格式

每行一条 JSON 记录，append-only 追加写入。类似于 Python 中的：

# 每次对话追加一行 with open("session.jsonl", "a") as f: json.dump({"role": "user", "content": msg}, f) f.write("\n")

为什么不用数据库？

本地优先原则：JSONL 文件可以直接用文本编辑器查看、用 Git 追踪变更、跨平台兼容，零运维成本。对于一个个人 Agent 来说，数据库是过度设计。

五、上下文压缩——长对话的生存机制

LLM 有 token 上限（如 128K tokens）。长对话会超出限制，必须"压缩"。

三级递进压缩策略

L1 LLM 摘要压缩

把最早的 50 条消息发给 LLM，让它生成一段摘要。50 条消息 ~30,000 tokens 压缩为 ~2,000 tokens（压缩 93%）

L2 工具结果截断

单条工具返回超过 500 tokens 时，只保留开头和结尾的关键部分

L3 旧消息移除

整体仍然超限时，从最早的消息开始移除，确保不超过窗口上限

六、记忆系统——跨会话的知识积累

会话内的记忆靠 JSONL，但跨会话的记忆需要专门的系统。

MEMORY.md

在工作区 .firmclaw/ 目录下维护一个结构化 Markdown 文件，记录用户偏好、技术决策、待办事项等。每条记忆有唯一 ID、内容、时间戳。

# MEMORY.md 示例 ## 偏好 [P001] 用户偏好 pnpm 而非 npm (2026-03-28) ## 技术决策 [T001] 项目使用 TypeScript strict 模式

BM25 全文搜索

纯 JavaScript 实现的搜索引擎（类似 Python 的 whoosh），支持中文分词。每次对话时自动检索相关记忆，注入到系统提示词中。

没有使用向量数据库或 embedding，而是用更轻量的关键词匹配。

七、安全机制——让 Agent 可控可信

HITL 人工审批

危险操作（如 rm -rf）在执行前暂停，等待用户确认。通过 Promise + 回调实现异步等待——类似 Python 的 input("确认执行？")。

GUARD Prompt Injection 防护

扫描工具返回结果中可能包含的注入攻击（如"忽略之前的指令"），用 [REDACTED] 替换可疑内容。不阻断执行，但净化输入。

AUDIT 审计日志

全量操作记录存储在 ~/.firmclaw/audit.jsonl（append-only），记录每次工具执行的操作类型、参数、风险等级、审批来源、执行结果和耗时。支持查询和 CSV 导出。

八、开发方法论——OpenClaw 的工程哲学

OpenClaw 教给我们的 5 条工程原则

1. 最小可用优先（MVP）

先做一个能跑的 ReAct 循环，再逐步叠加功能。不要一开始就设计完美的架构。

2. 渐进式架构（6 阶段）

每个阶段有明确的目标和验证标准。完成一个阶段再进入下一个，避免"大爆炸"式开发。

3. 本地优先（Local-First）

所有数据存本地文件，不依赖外部服务。用户对自己的数据拥有完全控制权。

4. 零外部依赖策略

核心功能只用 Node.js 内置模块。外部依赖（如 openai SDK）仅用于必须的 API 调用。

5. 可观测性优先

Agent 的每一步操作都应该能被追踪：事件流、审计日志、流式输出。

Chapter 03 — Core

ReAct 循环详解

Reasoning + Acting = ReAct。这是整个系统的心脏，一个 while 循环让 LLM 交替进行"思考"和"行动"。

Figure 4 — ReAct: Reasoning → Acting → Observation Loop

算法流程

1

初始化消息列表

构建 messages = [system_prompt, user_message]，system_prompt 定义智能体身份，user_message 是用户的问题。

2

调用 LLM（Reasoning）

将完整的 messages 数组和可用工具列表发送给 LLM。LLM 会根据上下文决定：直接回答，或者调用工具。

3

判断响应类型

检查 LLM 返回中是否包含 tool_calls。如果没有 → 直接返回文本，任务完成。如果有 → 进入步骤 4。

4

执行工具（Acting）

遍历每个 tool_call，解析参数 → 查找工具 → 执行 → 将结果以 role: 'tool' 加入 messages。

5

回到步骤 2

LLM 现在看到了工具执行结果，继续推理。直到 LLM 给出纯文本回复或达到 maxTurns 限制。

关键设计决策

决策理由
maxTurns 限制防止 LLM 陷入无限循环（如反复执行同一命令）
错误不中断工具失败时把错误文本反馈给 LLM，让它自行调整策略
tool_call_id 关联LLM 可能一次返回多个 tool_calls，通过 id 精确匹配结果
事件广播Agent Loop 不直接 console.log，通过 EventStream 实现解耦
Chapter 04

工具系统

工具是智能体的"手和脚"。任何工具只要实现 Tool 接口，就能被 Agent Loop 调用——零耦合，零改动。

工具接口

interface Tool {
  name: string         // LLM 通过这个名字调用
  description: string  // 告诉 LLM 工具能做什么
  parameters: ToolDef   // JSON Schema 参数定义
  execute: (params) => Promise<ToolResult>  // 执行逻辑
}

工具生命周期

Figure 6 — Tool Lifecycle: Register → Discover → Call → Result

Bash 工具的安全设计

  • ⏱ 30 秒超时 — 防止命令卡死导致智能体挂起
  • 📦 1MB 输出缓冲 — 防止命令输出过大撑爆内存
  • 🛡 错误不中断 — try/catch 捕获异常，返回错误文本给 LLM 自行调整
  • ✅ 参数校验 — 确认 command 参数存在且为字符串
Chapter 05

LLM Client

与 MiniMax M2.7 通信的封装层。使用 OpenAI 兼容格式，一套代码切换任意提供商。

四种消息角色

system

系统提示词，定义智能体身份和行为规则。由代码注入，用户不可见。

user

用户的输入消息。每次循环中 LLM 都能看到完整的用户消息。

assistant

LLM 的回复。可能包含纯文本，也可能包含 tool_calls（工具调用请求）。

tool

工具执行结果。通过 tool_call_id 关联到对应的 assistant 工具调用。

流式工具调用的累积过程

事件流

Figure 8 — EventStream: Pub/Sub Decoupling

事件类型

事件触发时机数据
thinking_deltaLLM 生成文本时（逐字）文本片段 string
tool_start工具开始执行工具名 + 参数
tool_end工具执行完成工具名 + 结果 + 是否错误
message_end最终回复完成完整文本
error出错时错误信息 string

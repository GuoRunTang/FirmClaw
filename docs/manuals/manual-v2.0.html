项目简介

FirmClaw 是一个从零搭建的 本地优先 AI 智能体框架，核心采用 ReAct（Reasoning + Acting）架构。灵感来源于 OpenClaw，目标是实现一个可理解、可扩展、本地运行的智能体系统。

一句话理解 FirmClaw

你在命令行输入一句自然语言，FirmClaw 会自动理解你的意图，调用合适的工具完成任务，然后把结果告诉你。它不是聊天机器人——它是能动手干活的 AI 助手。

系统架构

FirmClaw 采用经典的 ReAct 循环架构，各模块职责清晰、松耦合。

架构总览

ReAct 循环详解

  1. 用户输入 — 通过 CLI 传入自然语言指令
  2. LLM 推理 — Agent Loop 调用 LLM，LLM 决定是否需要工具
  3. 工具执行 — 如需工具，通过 ToolRegistry 执行（参数校验 → 权限检查 → 调用）
  4. 结果反馈 — 工具结果作为 tool 消息注入对话历史
  5. 继续思考 — LLM 看到结果后决定下一步（继续调工具 or 给出最终回复）
  6. 退出循环 — LLM 给出纯文本回复时，循环结束

Agent Loop v1.0

AgentLoop 是 FirmClaw 的心脏，实现了完整的 ReAct 推理-行动循环。

源文件：src/agent/agent-loop.ts

核心接口

interface AgentConfig v1.0 → v1.1
// src/agent/types.ts
systemPrompt: string // 系统提示词
maxTurns: number // 最大循环轮次
workDir?: string // 工作目录（v1.1 新增）
interface AgentResult
text: string // LLM 最终回复
turns: number // 总循环轮次
toolCalls: number // 总工具调用次数

循环流程

async run(userMessage: string): Promise<AgentResult> {
  messages = [system, user]
  toolContext = { workDir }

  while (turns < maxTurns) {
    // Step 1: 调用 LLM
    response = await llm.chat(messages, tools, onDelta)

    // Step 2: 无工具调用 → 返回最终结果
    if (!response.tool_calls) return result

    // Step 3: 执行所有工具
    for (toolCall of response.tool_calls) {
      result = await registry.execute(name, args, toolContext)
      messages.push({ role: 'tool', content: result })
    }
    // → 继续循环，LLM 看到工具结果后决定下一步
  }
}

v1.1 改进

  • 构建 ToolContext（包含 workDir），注入每次工具调用
  • 通过 registry.execute() 调用工具，自动完成参数校验
  • 不再直接调用 tool.execute()，校验逻辑解耦到 Registry 层

事件发射

事件触发时机data 内容
thinking_deltaLLM 生成文本时文本片段 string
tool_start工具开始执行{toolName, args}
tool_end工具执行完成{toolName, result, isError}
message_end最终回复完成完整文本
error发生错误错误信息 string

LLM Client v1.0

LLMClient 封装了与 LLM API 的通信层，支持 OpenAI 兼容格式，可无缝切换模型提供商。

源文件：src/llm/client.ts

核心消息格式

interface Message
role: 'system' | 'user' | 'assistant' | 'tool'
content: string
tool_call_id?: string // tool 消息关联 ID
tool_calls?: ToolCall[] // assistant 消息的工具调用
interface ToolCall
id: string // 调用唯一标识
type: 'function'
function.name: string
function.arguments: string // JSON 字符串

流式输出处理

LLM 的流式工具调用会分多次发送增量数据，LLMClient 自动累积拼接：

// 流式工具调用的数据流：
chunk 1: { tool_calls: [{ index: 0, id: 'xxx', function: { name: 'bash', arguments: '' } }] }
chunk 2: { tool_calls: [{ index: 0, function: { arguments: '{"co' } }] }
chunk 3: { tool_calls: [{ index: 0, function: { arguments: 'mmand":"ls"}' } }] }
// → 最终拼出: { id: 'xxx', type: 'function', function: { name: 'bash', arguments: '{"command":"ls"}' } }
模型无关设计：Agent Loop 不直接依赖 OpenAI SDK，只依赖 Message 接口。切换模型只需改 .env 配置，无需修改任何代码。

EventStream v1.0

EventStream 是智能体与外界通信的广播系统，基于 Node.js EventEmitter 实现。

源文件：src/utils/event-stream.ts

设计意图

Agent Loop 运行时，不同场景需要不同展示方式：

  • CLI — 实时输出 LLM 文本、工具调用进度
  • Web UI — 推送到 WebSocket
  • 测试 — 断言事件序列

EventStream 让 Agent Loop 与展示层完全解耦。

type AgentEventType

bash 工具 v1.0 → v1.5

智能体与操作系统交互的桥梁。v1.5 从 exec() 升级为 spawn()，大幅增强可靠性。

输出截断策略

当输出超过 100KB 时，自动截断并标注：

[Output truncated: 256KB total, showing first 100KB]
安全提示：bash 工具受权限策略保护。危险命令（rm -rf /、format、shutdown 等）会被黑名单拦截。

权限策略系统 v1.6

三层安全防护，作为中间件层不侵入工具实现。

源文件：src/tools/permissions.ts

三层防护架构

接口定义

interface PermissionPolicy
checkFileAccess?(path, op) → PermissionResult // 校验文件权限
checkCommand?(cmd) → PermissionResult // 校验命令权限

默认实现：DefaultPermissionPolicy

class DefaultPermissionPolicy implements PermissionPolicy
allowedPaths: string[] // 允许的目录（默认 [workDir]）
extraAllowedPaths: string[] // 额外允许路径
commandBlacklist: string[] // 危险命令列表

默认黑名单

// bash 命令黑名单（部分匹配）
['rm -rf /', 'rm -rf /*', 'del /f /s /q c:\\', 'format ',
 'diskpart', 'shutdown ', 'taskkill /f /im',
 ':(){ :|:& };:',  // fork bomb
 'mkfs.', 'dd if=']

// 敏感文件（禁止写入）
['.env', '.env.local', '.env.production',
 'credentials.json', 'id_rsa', 'id_ed25519', '.ssh']

// 系统保护目录（Windows）
['C:\\Windows', 'C:\\Program Files',
 'C:\\Program Files (x86)', 'C:\\ProgramData']

Phase 5 扩展性

PermissionPolicy 接口可直接扩展为人工审批（Human-in-the-Loop）：

class HumanApprovalPolicy implements PermissionPolicy {
  async checkFileAccess(path, op): Promise<PermissionResult> {
    // 弹出确认对话框，等待用户审批
    const approved = await askUser(`Allow ${op} on ${path}?`);
    return { allowed: approved, reason: approved ? undefined : 'User rejected' };
  }

async run(userMessage) {
  // 1. 构建系统提示词（支持 query 搜索相关记忆）
  systemPrompt = contextBuilder.build(tools, meta, userMessage);

  // 2. 恢复历史消息

    historyMessages = result.messages;

  }

  // 4. TokenCounter 简单裁剪（保底）
  if (tokenCounter) { messages = tokenCounter.trimMessages(allMessages); }

  // 5. ReAct 循环...
}

TokenCounter v2.3

Token 计数器和上下文裁剪器，当摘要压缩不足时的保底措施。

源文件：src/utils/token-counter.ts

裁剪策略

  • 从最早的消息开始裁剪
  • 优先裁剪 tool 结果（保留决策上下文）
  • 始终保留 system prompt 和最近 2 条消息

Summarizer v3.1

LLM 摘要压缩器 —— 将旧消息压缩为高质量摘要，保留关键决策和语义信息。

源文件：src/session/summarizer.ts

设计要点

  • 复用 LLMClient：不引入新依赖，复用已有的 LLM 通信层
  • 滑动窗口：摘要成为新的"锚点"，后续基于摘要+新消息继续压缩
  • 容错设计：摘要失败不阻塞主流程，返回原始消息
  • 摘要 prompt：明确要求保留关键决策(✓)、用户偏好、技术方案、待办(○)

接口定义

class Summarizer
// 构造函数
constructor(llm: LLMClient, tokenCounter: TokenCounter, config?: SummarizerConfig)

// 判断是否需要摘要
shouldSummarize(messages: Message[]): boolean

// 执行摘要压缩
async summarize(messages: Message[]): Promise<SummaryResult>

配置参数

summarizeThreshold number 触发摘要的 token 阈值，默认 80000
maxMessagesToSummarize number 每次最多压缩的消息条数，默认 50
maxSummaryTokens number 摘要最大 token 数，默认 2000
verbose boolean 是否输出摘要统计到控制台

摘要流程

1. shouldSummarize(messages)
   → 检查历史消息 token 数是否 >= 80000

2. summarize(messages)
   → 找到最早的可压缩消息段（跳过已有摘要）
   → 取最多 50 条消息，保留最后 1 条
   → 构建摘要请求 → 调用 LLM
   → 用 [摘要] system 消息替代原始消息段
   → 返回 { messages, summarized, compressedCount, originalTokens, newTokens }

使用示例

// index.ts 中的初始化
const summarizer = new Summarizer(llm, tokenCounter, {
  summarizeThreshold: 80000,
  maxMessagesToSummarize: 50,
});

// AgentLoop 自动调用（无需手动触发）
const agent = new AgentLoop(llm, tools, { ..., summarizer });

// 手动触发：/compact 命令
if (summarizer.shouldSummarize(messages)) {
  const result = await summarizer.summarize(messages);
  // result.newTokens < result.originalTokens
}
注意：摘要需要一次额外的 LLM API 调用，会增加约 2-5 秒延迟和少量 token 消耗。仅在历史消息超过 80K tokens 时自动触发。

SearchEngine v3.3

纯 JS BM25 全文搜索引擎，零外部依赖，支持中英文混合分词和索引持久化。

源文件：src/session/search-engine.ts

BM25 算法

score(D, Q) = Σ IDF(qi) × (f(qi,D) × (k1+1)) / (f(qi,D) + k1 × (1-b + b × |D|/avgdl))

参数：k1 = 1.5（词频饱和），b = 0.75（长度归一化）
IDF  = ln((N - n + 0.5) / (n + 0.5) + 1)  // Okapi BM25 变体

中英文混合分词策略

tokenize("FirmClaw 使用 ReAct 架构")
→ ["firmclaw", "使用", "使", "用", "用r", "re", "ea", "ac", "ct", "re", "ea", "ac", "ct", "架", "架构", "构", "react"]
  └─ 英文单词（转小写）    └─ 中文 bigram + 单字

使用示例

await engine.load();

// 添加文档

  id: 'doc-1',
  source: 'session',
  content: 'FirmClaw 使用 ReAct 架构实现智能体循环',
  timestamp: '2026-03-29T10:00:00Z',

// 搜索

// → [{ id: 'doc-1', score: 8.52, snippet: '...ReAct 架构实现...', source: 'session' }]

// 搜索记忆条目
const memoryIds = engine.searchMemory('pnpm', memoryEntries, 5);

// 持久化到磁盘
await engine.persist();
// → ~/.firmclaw/index/search-index.json
纯 JS 实现：不依赖 SQLite / better-sqlite3 / sqlite-vec。索引存储在内存中，退出时序列化为 JSON。搜索 1000 文档 < 50ms。

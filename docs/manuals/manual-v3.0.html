项目简介

FirmClaw 是一个从零搭建的 本地优先 AI 智能体框架，核心采用 ReAct（Reasoning + Acting）架构。v3.0 整合了 Phase 1（ReAct 循环）、Phase 2（工具系统）和 Phase 3（会话管理与上下文管理）的全部能力。

一句话理解 FirmClaw

你在命令行输入一句自然语言，FirmClaw 会自动理解意图、调用工具完成任务，并记住之前的对话。它不是聊天机器人——它是有记忆、能动手干活的 AI 助手。

Agent Loop Phase 1 + Phase 3

AgentLoop 是 FirmClaw 的心脏。v2.3 集成了会话管理、动态提示词和上下文裁剪，但保持了向后兼容——所有 Phase 3 组件均为可选。

源文件：src/agent/agent-loop.ts

核心接口（v2.3 更新）

interface AgentConfig v2.3 新增字段
systemPrompt: string // 固定提示词（contextBuilder 未设置时使用）
maxTurns: number // 最大循环轮次
workDir?: string // 工作目录
sessionManager?: SessionManager // 会话管理器（v2.1）
contextBuilder?: ContextBuilder // 提示词组装器（v2.2）
tokenCounter?: TokenCounter // Token 计数器（v2.3）
trimConfig?: TrimConfig // 裁剪配置（v2.3）

v2.3 循环流程

async run(userMessage: string): Promise<AgentResult> {
  // ① 自动创建会话（如果未启用）
  if (sessionManager && !currentSession) {
    meta = await sessionManager.create(workDir, userMessage)
  }

  // ② 动态生成系统提示词
  if (contextBuilder) {
    systemPrompt = await contextBuilder.build(tools, sessionMeta)
  }

  // ③ 恢复历史消息
  history = await sessionManager.getMessages()
  messages = [system, ...history, user]

  while (turns < maxTurns) {
    // ④ 上下文裁剪（token 超限时）
    if (tokenCounter) tokenCounter.trimMessages(messages, trimConfig)

    // ⑤ 调用 LLM（ReAct 核心不变）
    response = await llm.chat(messages, tools, onDelta)

    // ⑥ 无工具调用 → 保存会话 → 返回
    if (!response.tool_calls) {
      await persistRound(userMessage, roundMessages)
      return result
    }

    // ⑦ 执行工具（v1.0 逻辑不变）
    for (toolCall of response.tool_calls) {
      result = await registry.execute(name, args, toolContext)
      messages.push({ role: 'tool', content: result })
      roundMessages.push(toolMsg)  // 收集本轮消息
    }
  }
}

会话持久化策略

  • 自动创建：首次发送消息时自动创建会话（标题 = 用户第一条消息的前 50 字）
  • 批量保存：一轮循环结束后，一次性 appendBatch 保存所有消息（user + assistant + tool）
  • 异步元数据更新：消息计数和更新时间异步写入磁盘，不阻塞主流程
  • 恢复机制：启动时 resumeLatest() 自动恢复上次会话

三层防护架构

SessionStore Phase 3

JSONL 存储层，负责会话消息的磁盘读写。设计为 append-only，不修改历史行。

源文件：src/session/store.ts

JSONL 文件格式

#META {"id":"a1b2c3d4","createdAt":"2026-03-29T...","title":"帮我分析代码","messageCount":5}
{"role":"user","content":"帮我分析代码","timestamp":"2026-03-29T10:00:01.000Z"}
{"role":"assistant","content":"好的，让我来读取文件...","timestamp":"2026-03-29T10:00:05.000Z"}
{"role":"tool","content":"File: src/index.ts (lines 1-50 of 100)...","tool_call_id":"call_abc","timestamp":"..."}

设计要点

  • Append-only：新消息只追加到文件末尾，不修改历史行
  • 原子写入：使用 fs.appendFile，操作系统保证追加的原子性
  • 懒加载：readMessages() 时才读取文件，不在内存中缓存全量
  • 路径安全：sessionId 使用正则 /^[a-zA-Z0-9_-]+$/ 校验，防止路径遍历攻击
  • 容错读取：无法解析的 JSON 行会被跳过（不中断整个读取）

核心方法

方法说明
create(meta)创建新会话文件，写入 #META 首行
append(sessionId, message)追加单条消息
appendBatch(sessionId, messages)批量追加（一轮循环的多条消息）
readMessages(sessionId)读取全部消息（跳过 #META）
readMeta(sessionId)读取会话元数据
updateMeta(sessionId, meta)更新元数据（替换首行，保留消息）
listAll()列出所有会话（按 updatedAt 降序）
delete(sessionId)删除会话文件

SessionManager Phase 3

会话管理器，封装 SessionStore，对外提供面向业务的 API。Agent Loop 和 CLI 都通过它与会话系统交互。

源文件：src/session/manager.ts

核心接口

interface SessionMeta
id: string // crypto.randomUUID() 格式
createdAt: string // ISO 8601
updatedAt: string // ISO 8601
workDir: string // 关联的工作目录
title: string // 用户第一条消息的前 50 字
messageCount: number // 消息条数
interface StoredMessage
role: 'system' | 'user' | 'assistant' | 'tool'
content: string
tool_call_id?: string
tool_calls?: ToolCall[]
timestamp: string // ISO 8601（FirmClaw 内部字段，不传给 LLM）

核心方法

方法说明
create(workDir, firstMessage?)创建新会话（自动生成 ID 和标题）
resume(sessionId)恢复指定会话（读取元数据并设为当前）
resumeLatest()恢复最近一次会话（按 updatedAt 排序）
append(messages)向当前会话追加消息（异步更新 meta）
getMessages()获取当前会话的 LLM 消息数组（StoredMessage → Message）
listSessions()列出所有会话（按 updatedAt 降序）
gc(maxAgeDays?)清理过期会话（默认 30 天）
switchSession(sessionId)切换到指定会话（不读取历史）

关键设计

  • metaCache：内存中缓存元数据，避免频繁读文件。append() 时异步更新磁盘，不阻塞主流程
  • toLLMMessage：内部函数将 StoredMessage 转为 Message（剥离 timestamp），因为 LLM 不需要时间戳
  • 禁用模式：config.enabled = false 时所有操作变为 no-op，便于测试

TokenCounter Phase 3

Token 估算与消息裁剪器。当对话过长可能超出 LLM 上下文窗口时，TokenCounter 会自动裁剪消息。

源文件：src/utils/token-counter.ts

估算策略

约 4 字符 ≈ 1 token（中英文混合场景的粗略估计）。这不是精确计数，但足以做裁剪判断。Phase 4 可升级为 tiktoken。

两级裁剪策略

裁剪配置

interface TrimConfig

裁剪结果

interface TrimResult

trimmedTokens: number // 裁剪后总 token 数
